In this lab you will explore real weather metrics collected by the TA-open-meteo add-on, compare events vs. metrics performance, and convert event data into a metrics index using `mcollect`.

### Exercise 3: Events vs. Metrics — Performance Comparison

This is the key lesson: the **same data** queried from an event index vs. a metric index. Run both searches and compare the execution time in the Job Inspector.

**Events-based search (slower):**
<SplunkSearch earliest="-7d" latest="now" spl={`index=s4c_meteo
| stats avg(temperature_2m) as avg_temp by city
| sort - avg_temp`}></SplunkSearch>

**Metrics-based search (faster):**
<SplunkSearch earliest="-7d" latest="now" spl={`| mstats avg(weather.temperature_2m) as avg_temp WHERE index=s4c_meteo_metrics BY city
| sort - avg_temp`}></SplunkSearch>

Open the **Job Inspector** (the "i" icon next to each search result) and compare:
- **Total execution time** — mstats should be significantly faster
- **Scanned events / data points** — metrics scans are more efficient
- **Disk usage** — metrics indexes are optimized for numeric time-series storage

Why is `mstats` faster? Metric indexes store pre-extracted numeric values with dimensions already indexed. There is no need for field extraction, regex parsing, or full-text search at query time.

### Exercise 4: Convert Events to Metrics with mcollect

This exercise teaches you how to take existing event data and convert it into metric data points. This is exactly what the TA-open-meteo add-on does automatically — but here you will do it manually to understand the process.

**Step 1: Preview the event data you will convert**

First, look at the raw events in `s4c_meteo` to understand the field structure:
<SplunkSearch earliest="-1h" latest="now" spl={`index=s4c_meteo
| head 5
| table _time city country temperature_2m relative_humidity_2m wind_speed_10m precipitation`}></SplunkSearch>

**Step 2: Build the conversion search**

This search selects weather fields from the event index and prepares them for metric conversion:
<SplunkSearch earliest="-1h" latest="now" spl={`index=s4c_meteo earliest=-1h
| fields _time city country temperature_2m relative_humidity_2m wind_speed_10m precipitation`}></SplunkSearch>

**Step 3: Run the mcollect conversion**

Remove the backtick comments from the `mcollect` line below to actually write the data. This will create metric data points in `s4c_student_metrics` (a separate index so you do not duplicate data in the production metrics index).

<SplunkSearch earliest="-1h" latest="now" spl={`index=s4c_meteo earliest=-1h
| fields _time city country temperature_2m relative_humidity_2m wind_speed_10m precipitation
\`\`\`| mcollect index=s4c_student_metrics prefix_field=city\`\`\`
`}></SplunkSearch>

Remove the comments from:
`| mcollect index=s4c_student_metrics prefix_field=city`
once the index `s4c_student_metrics` is ready and you want the data to be written.

**What mcollect does:**
- Takes each row from the search results
- Converts numeric fields (`temperature_2m`, `relative_humidity_2m`, etc.) into metric name/value pairs
- Uses `prefix_field=city` to prefix metric names with the city dimension
- Writes the resulting metric data points to the target index

**Step 4: Verify your converted metrics**

After running the mcollect, check that your metrics arrived:
<SplunkSearch earliest="-1h" latest="now" spl={`| mcatalog values(metric_name) values(city) WHERE index=s4c_student_metrics`}></SplunkSearch>

Query your newly created metrics:
<SplunkSearch earliest="-1h" latest="now" spl={`| mstats avg(temperature_2m) as avg_temp WHERE index=s4c_student_metrics BY city`}></SplunkSearch>

### Why Metrics Indexes Matter

Think about why a dedicated metrics index is better for time-series data:

1. **Performance**: `mstats` queries on metrics indexes are dramatically faster than `stats` on event indexes because numeric values are pre-extracted and indexed.
2. **Storage efficiency**: Metrics indexes store only the numeric values and dimensions — no raw text, no `_raw` field, no full-text index.
3. **Scalability**: As data volumes grow, metrics indexes scale much better for aggregation queries (avg, sum, max, min, count).
4. **Purpose-built**: The `mstats`, `mcatalog`, and `mpreview` commands are optimized for metric data patterns.

The TA-open-meteo add-on automates this dual-write pattern: events for ad-hoc investigation, metrics for performant dashboards and alerting.
