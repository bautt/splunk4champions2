## Tokens from inputs
Tokens from inputs work the same way like in xml dashboards. Inputs can be used above and inside canvas. Inputs with no data available can be hidden. 
* <Link to="https://docs.splunk.com/Documentation/SplunkCloud/latest/DashStudio/inputConfig" openInNewContext> "Adding and configuring inputs" </Link>

## Tokens from search
Dashboard Studio has the ability to reference any search results directly as a token. The syntax used is "$data source name:result.field$"
* <Link to="https://docs.splunk.com/Documentation/SplunkCloud/latest/DashStudio/searchTokens" openInNewContext> "Setting tokens from search results or search job metadata" </Link>

In order to be able to use search results and job properties as tokens, you have to check the "Access search results or metada" checkbox in the data source configuration.

## Drilldown tokens and conditional hiding
Since version 9.1 (Cloud 9.0.2303) you can hide panels in the dashboard if the search has no results. This feature is called conditionally visible panels. 
The logic behind it can also be used to make panels dependent on a particular token. If the token is not set to a defined value the search would return no results and the panel will be hidden.

## Button Input and Token Reset (New in Splunk 10.0)

Splunk 10.0 introduced a native **Button input** (`input.button`) in Dashboard Studio. Buttons set token values when clicked — no custom JavaScript required. A Reset button can return the dashboard to its default state.

### Configuring a Button

```json
"input_btn_web": {
  "type": "input.button",
  "options": { "label": "Web" },
  "eventHandlers": [
    {
      "type": "drilldown.setToken",
      "options": {
        "tokens": [
          { "token": "category_filter", "value": "web" }
        ]
      }
    }
  ]
}
```

A **Reset button** uses the same mechanism — it simply sets the token back to its default value (`"all"`):

```json
"input_btn_reset": {
  "type": "input.button",
  "options": { "label": "Reset" },
  "eventHandlers": [
    {
      "type": "drilldown.setToken",
      "options": {
        "tokens": [ { "token": "category_filter", "value": "all" } ]
      }
    }
  ]
}
```

Set default token values in the `tokens` section so the dashboard renders correctly on first load:

```json
"tokens": {
  "defaultTokenValues": {
    "category_filter": { "value": "all" }
  }
}
```

## Conditional Panel Visibility with expressions

In Splunk 10.0, panels can be shown or hidden based on **named conditions** evaluated against token values. This replaces the older search-result-empty approach with a cleaner, more readable syntax.

### Define conditions in the `expressions` section:

```json
"expressions": {
  "conditions": {
    "cond_is_web": {
      "name": "show web",
      "value": "$category_filter$ = \"web\""
    },
    "cond_is_all": {
      "name": "show all",
      "value": "$category_filter$ = \"all\""
    }
  }
}
```

### Reference conditions in `containerOptions.visibility`:

```json
"viz_web_panel": {
  "type": "splunk.line",
  "dataSources": { "primary": "ds_web_data" },
  "title": "Web Server: Request Trend",
  "containerOptions": {
    "visibility": {
      "showConditions": ["cond_is_web", "cond_is_all"]
    }
  }
}
```

The panel is visible when **any** of the listed conditions is true. Panels not matching any condition are hidden — no data is fetched for hidden panels, so this also improves performance.

### Complete Pattern: Button Filter with Conditional Panels

1. Add `input.button` inputs to `globalInputs`
2. Each button sets a `category_filter` token via `drilldown.setToken`
3. Define named conditions in `expressions.conditions`
4. Each panel references the relevant conditions in `containerOptions.visibility`
5. The Reset button restores the token to `"all"` so all panels reappear
