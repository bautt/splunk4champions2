## SPL2 Module Editor

The **Module Editor** is the primary SPL2 interface in the Search & Reporting app. Unlike the classic search bar — which runs one search at a time — the Module Editor lets you write, name, and run **multiple searches within a single file**, switching between their results without leaving the page.

Think of it as a Jupyter notebook for Splunk: your searches, documentation, and results all in one place.

---

### How to Access

1. Open **Search & Reporting** from the Splunk home screen
2. Click the **Modules** tab (next to Search and Dashboards)
3. Click **New module**

Or from the Search page: select **Get started with guided help** for an interactive tutorial.

---

### Module Editor Interface

The editor has eight components:

| Component | Purpose |
|---|---|
| **Outline panel** | Lists all named searches in the module for quick navigation; add new searches here |
| **SPL2 panel** | Write your searches, commands, and functions |
| **Search results panel** | Displays the output of the currently selected search |
| **Time range picker** | Sets the time window (default: Last 24 hours) |
| **Run icon** | Executes the selected search (also `Ctrl+Enter` / `Cmd+Enter`) |
| **Data list** | Browse and select datasets — indexes, data views, lookups |
| **Actions panel** | Point-and-click data manipulation; updates your SPL2 automatically |
| **Fields panel** | Field metrics and summaries for the current result set |

---

### What Modules Enable

**Multi-statement investigations**

A module contains multiple named searches. You run them individually, in sequence, or conditionally — each building on previous results. This is particularly powerful for incident response: work from alert → investigation → root cause → documentation without switching tabs or losing context.

<img src="https://www.splunk.com/content/dam/splunk-blogs/images/media_1c635ae4e16b1b1b978f49af268e151c8f86b5fba.avif" width="100%" alt="SPL2 Module Editor showing incident analysis with chained searches"/>

**Named search branching**

Name a search result, then reference it in downstream searches. Instead of rerunning the same base query multiple times, you define it once and branch from it — similar to base/chain searches in Dashboard Studio, but within a single investigation workflow.

**Inline documentation**

Add comments and notes directly alongside your searches. When you hand the module to another analyst or close the ticket, the thought process is already embedded in the file.

**Collaboration and sharing**

Modules can be exported and shared across teams with configurable permissions. Other analysts can import your module, build on top of your searches, and extend the investigation — creating a single source of truth for incident closure.

---

### Data Views

SPL2 introduces **data views** — admin-defined, permissioned projections over indexes. Instead of granting users access to an entire index, an admin creates a view that exposes only specific fields and time ranges.

<img src="https://www.splunk.com/content/dam/splunk-blogs/images/media_13234e04f9c37ffa76f73ad22069ef6d9f66295a4.avif" width="100%" alt="SPL2 data views — create and export permissioned views over indexes"/>

Users search against the view name using `FROM view_name` — the underlying index and its full data remain protected.

This enables:
- Sharing data with teams who only need a subset of an index
- Enforcing field-level access control without complex transforms
- Creating purpose-built datasets for dashboards or external teams

---

### Keyboard Shortcuts

| Shortcut | Action |
|---|---|
| `Ctrl+Enter` / `Cmd+Enter` | Run selected search |
| `Ctrl+Space` | Trigger autocomplete / help |
| `Ctrl+/` | Comment / uncomment line |
| `Ctrl+F` | Find terms in the editor |

---

### SPL Embedding

If you have existing SPL1 searches you want to keep, SPL2 supports **embedding legacy SPL** within an SPL2 pipeline. This allows gradual migration — wrap your existing search in SPL2 syntax without rewriting it, and incrementally modernize over time.
